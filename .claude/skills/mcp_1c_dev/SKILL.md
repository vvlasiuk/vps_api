# Розробка запитів 1С (mcp_1c_dev)

Правила роботи з MCP-сервером `mcp_1c_dev` і каталогом `queries1c/` у `vps_api`.
Сервер дає читання структури конфігурації 1С і збереження іменованих запитів
(`.sel` + `.json`). **Виконання запитів і доступ до бойових даних сюди не входять**
(це окремий сервер `mcp_1c_read`).

## Інструменти сервера

Читання (безпечне, нічого не змінює):
- `list_objects()` → `{total, objects:[{type, name, synonym}]}` — усі довідники + документи.
- `describe_object(object_type, object_name)` → `{type, name, synonym, attributes[], tabular_sections[]}`.
  `object_type` = `"Справочник"` | `"Документ"`.
- `list_queries(object_type, object_name)` → `{total, queries:[{query_name, info, file, fields_count}]}`.
  Тільки поштучно по одному об'єкту — глобального режиму немає.
- `get_query(query_name)` → `{query_name, file, sel, meta}`. `file` — реальне ім'я файлу без розширення.

Розробка:
- `generate_query(object_type, object_name)` → `{sel, meta}` — механічна болванка з опису
  об'єкта. **На диск НЕ пише, AI не використовує.** Системні `_*` поля згори (6 для довідника,
  5 для документа), решта реквізитів з мапінгом типів, аліас `дов`/`док`, `source_name`.
  Це основа: болванку треба почистити (див. нижче), а не зберігати як є.
  Якщо в сесії інструмент недоступний — збери таку болванку вручну за тією ж формою.
- `save_query(sel, meta, file_name="")` → `{ok, query_name, path_sel, path_json, total_queries}` —
  запис `.sel` + `.json` на диск + гарячий перечит loader (без рестарту, зміна одразу "жива").

## Робочий цикл (запис автономний)

Задачу ставить користувач; далі дію сам, без запиту дозволу на запис. Показую diff
**після** збереження.

**Новий запит:**
1. `describe_object` — отримати реальні реквізити, типи, ref-цілі, табличні частини.
2. `generate_query` — взяти болванку як основу.
3. Почистити: прибрати зайві поля, задати `snake_case`-аліаси (див. конвенції),
   поставити `query_name` та фільтри під задачу.
4. **Верифікація** (обов'язково, див. нижче).
5. `save_query` — зберегти, потім показати diff.

**Редагування наявного:**
1. `get_query(query_name)` — взяти поточні `sel`/`meta` і **реальне `file`**.
2. Внести зміни в межах задачі.
3. Верифікація.
4. `save_query(..., file_name=<file з get_query>)` — інакше створиться дубль.

## Обов'язкова верифікація перед кожним записом

Перед будь-яким `save_query` спершу `describe_object` відповідного об'єкта і звірити:
- кожне поле `sel` існує на об'єкті (з урахуванням кирилиця/латиниця в іменах);
- `type` у `meta.fields` збігається з типом реквізиту (`Рядок`→string, `Число`→number,
  `Булево`→boolean, `Дата`→date, ref→ref);
- ref-поле вказує на очікуваний об'єкт;
- dot-шляхи (`Інструмент.ПСТ_Бренд` тощо) валідні по ланцюгу;
- фільтри по неявних полях (`Родитель.Код`, `Підрозділ.Код`) посилаються на наявні реквізити;
- поле в шапці, а не в табличній частині (табличні — окремо).

Один і той самий об'єкт у межах розмови звіряти достатньо раз.

**Пастка `generate_query`:** болванка **завжди** додає довіднику 6 системних полів,
включно з `_parent`/`_isfolder`. Якщо довідник **неієрархічний** (у `describe_object`
немає `Родитель`/`ЭтоГруппа`) — ці два поля треба прибрати і з `sel`, і з `meta`.

## Конвенції нових запитів

- **Аліаси — `snake_case` латиницею.**
  - Системні: `_ref`, `_code`, `_description`, `_marked`, `_parent`, `_isfolder` (довідник);
    `_ref`, `_version`, `_number`, `_date`, `_status` (документ).
  - Змістовні — англо-доменні: `name`, `code`, `brand_ref`, `contractor`, `tool_type` і т.д.
    (не кирилиця, не імена реквізитів 1С).
- **`source_name: "1C_UTP"`** — у `meta` кожного запиту.
- **Аліас таблиці:** `дов` для довідника, `док` для документа.
- **Ієрархічні довідники:** у `ГДЕ` завжди `НЕ <об>.ПометкаУдаления И НЕ <об>.ЭтоГруппа`
  (для неієрархічних — лише `НЕ ПометкаУдаления`).
- **`_status` для документів** — вираз, а не поле:
  ```
  ВЫБОР
      КОГДА док.Проведен      ТОГДА 1
      КОГДА док.ПометкаУдаления ТОГДА 2
      ИНАЧЕ 0
  КОНЕЦ КАК _status
  ```
  (0 = не проведено, 1 = проведено, 2 = помічено на вилучення).
- `meta.fields` мусить один-в-один відповідати полям у `sel` (той самий склад і порядок).

## Незмінний стиль правок

- **Не чіпати коментарі й закоментований код** — навіть стале «// ТЕСТОВА ГЕНЕРАЦІЯ»
  чи «// Чернетка»; прибирати тільки за прямою вказівкою.
- **Зміни лише в межах задачі.** Жодних непроханих покращень сусідніх полів/запитів.
- **Повний файл — коли просять**; інакше точкова правка.
- Не додавати поля, яких користувач не просив (правило «байт-у-байт» для режиму редагування).

## Механіка й пастки

- **`file_name` при редагуванні = реальне `file` з `get_query`**, не `query_name`.
  Приклад: запит `cat_contractors` лежить у файлі `Контрагенты`; запис із
  `file_name="cat_contractors"` створив би дубль.
- **`list_queries`** — лише по одному об'єкту за виклик.
- **Розкладка файлів:** `queries1c/catalogs/<ObjectName>/` та `queries1c/documents/<ObjectName>/`;
  ключ реєстрації — `query_name` з `.json`.
- Суміжне (сервер `mcp_1c_read`, виконання запитів): парсинг відповіді
  `json.loads(outer[0]['text'])` → `total`/`rows`; надійний розмір сторінки `limit=1000`
  (`5000` падає).
