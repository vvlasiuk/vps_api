# Інструкції проєкту: Розробка vps_api

Ти допомагаєш розробляти систему vps_api — FastAPI-бекенд + статичні HTML-сторінки,
що працюють із 1С 8.3 (УТП) через проміжний HTTP-сервіс. Інтерфейс українською.
У тебе є MCP-інструменти для доступу до системи (див. розділ «MCP» нижче) — користуйся
ними, щоб брати реальний контекст, а не вигадувати.

## Головні принципи роботи

- Аналізуй мовчки перед дією; роби мінімальні, точкові зміни строго в межах задачі.
- Не чіпай коментарі й закоментований код, якщо про це не просять.
- Кожен блок змін оформлюй як цикл: **бекап → зміни → лог** (див. «Цикл блоку змін»).
- Відповідай стисло й по суті. Українською.
- Не вигадуй структуру 1С, поля чи API компонентів — бери їх через MCP (describe_object,
  get_query, read_form).
- Зміни в скриптах/бекенді вносить користувач сам. Твоя роль — давати точні готові
  фрагменти/повні файли, а не застосовувати їх автономно.

## Цикл блоку змін (бекап → зміни → лог)

Будь-яка зміна на диску (запис запитів або форм) — це блок, оформлений так:

1. **Бекап** — ПЕРЕД змінами запропонуй/зроби create_backup (відкриваюча дужка).
2. **Зміни** — save_query та/або write_form (скільки треба для однієї команди).
3. **Лог** — наприкінці ОДИН виклик log_command на всю команду (закриваюча дужка).

Правила логу:
- Одиниця логу — **команда користувача**, не окрема операція запису. Одна команда, що
  зачепила і .sel/.json, і html — це ОДИН log_command із усіма файлами в полі files.
- log_command — останній крок; якщо забути його, пропуск буде тихим (ніщо не нагадає).
  Тому це обов'язкова закриваюча дужка блоку, парна до бекапу.
- Поля: cmd (суть команди укр., обов'язкове), desc (короткий ASCII для імені файлу,
  обов'язкове), clar (уточнення з діалогу), why (мотив, якщо був), files (зачеплені
  файли, шляхи від кореня проєкту). Порожній cmd або desc → помилка, файл не створюється.
- user і час проставляє сервер (з токена сесії й системного годинника) — ти їх не передаєш.

## Канали доступу (важливо)

- **MCP-інструменти** — єдиний канал ЗАПИСУ на диск: save_query (запити), write_form
  (форми, лише pages/ та menu/), create_backup (бекап), log_command (журнал команд).
- **Filesystem** — ТІЛЬКИ ЧИТАННЯ. Використовуй його, щоб дивитися структуру/файли
  проєкту як контекст. НІЧОГО не пиши через Filesystem — жодних логів, форм, запитів.
  Зокрема журнал команд пишеться ВИКЛЮЧНО через log_command, не через Filesystem
  (єдине джерело правди щодо формату — бекенд).

## Архітектура (коротко)

HTML (компонент) → FastAPI (/1c/query, /1c/save_doc) → 1С HTTP-сервіс (hs/vps/...) → 1С

- Дані читаються через ІМЕНОВАНІ запити (queries1c/), не сирим текстом.
- Фронт працює з АЛІАСАМИ полів (латиниця), не знаючи структури 1С.
- Бекенд обгортає запит: ВЫБРАТЬ {fields} ИЗ ( <текст .sel> ) КАК Вложенный,
  і накладає зверху fields/filters/order по аліасах.

## Система запитів (.sel + .json)

Кожен запит — пара файлів у queries1c/<catalogs|documents>/<Обʼєкт>/:
- .sel — текст запиту 1С (російські ключові слова: ВЫБРАТЬ, ИЗ, ГДЕ, КАК). // — коментар.
- .json — метадані. Джерело правди. Структура:
  {
    "query_name": "<ASCII-ідентифікатора>",    // напр. cat_contractors_dropdown
    "object_type": "<Справочник|Документ>",
    "object_name": "<імʼя обʼєкта>",
    "source_name": "1C_UTP",
    "info": "<опис українською>",
    "fields": [ {"key":"<аліас>", "type":"<тип>", "info":"<опис>"} ]
  }

Порядок ключів у .json саме такий (query_name згори, fields внизу).

Типи полів (fields[].type) — СУВОРО одне з: ref, string, number, date, boolean.
  посилання (СправочникСсылка/ДокументСсылка/...) → ref
  Строка → string, Число → number, Дата → date, Булево → boolean

### Системні поля (обовʼязково, згори .sel і fields)

Псевдонім таблиці: дов (довідник), док (документ).

Довідник — завжди:
  дов.Ссылка КАК _ref (ref), дов.Код КАК _code (string),
  дов.Наименование КАК _description (string), дов.ПометкаУдаления КАК _marked (boolean)
Довідник — ДОДАТКОВО лише якщо ієрархічний (є Родитель/ЭтоГруппа в describe):
  дов.Родитель КАК _parent (ref), дов.ЭтоГруппа КАК _isfolder (boolean)

Документ — завжди:
  док.Ссылка КАК _ref (ref), док.ВерсияДанных КАК _version (string),
  док.Номер КАК _number (string), док.Дата КАК _date (date),
  ВЫБОР КОГДА док.Проведен ТОГДА 1 КОГДА док.ПометкаУдаления ТОГДА 2 ИНАЧЕ 0 КОНЕЦ КАК _status (number)

Спершу системні поля, далі прикладні. Аліаси прикладних — осмислена латиниця
(name, code, edrpou, brand_ref). Для поля-посилання, по якому фільтруватимуть,
виводь окремий аліас (напр. дов.ПСТ_Бренд КАК brand_ref).
Для довідників типово додавай ГДЕ НЕ дов.ПометкаУдаления.

ВАЖЛИВО: для генерації болванки використовуй інструмент generate_query (він дає
коректні системні поля, фільтрацію стандартних, source_name). Не відтворюй це вручну —
візьми болванку й допрацюй під задачу.

### Режими виконання .sel

Роутер `/1c/query` (`onec_router.py`) обгортає текст `.sel` у зовнішній
`ВЫБРАТЬ`, щоб застосувати проєкцію (`fields`), відбір (`filters`) і
сортування (`order`). Роутер розпізнає режим автоматично за вмістом `.sel`.

**Режим 1 (типовий) — один ВЫБРАТЬ.** Пиши `.sel` як один запит; роутер
обгортає його підзапитом `ИЗ ( <текст .sel> ) КАК Вложенный`. `{fields}`,
`{filters}`, `{order}` посилаються на аліаси верхнього ВЫБРАТЬ твого .sel —
це і є публічний контракт виходу. Обмеження: увесь `.sel` іде як один
підзапит усередині `ИЗ (...)` — кілька пакетів (`;`) і temp-таблиці
(`ПОМЕСТИТЬ`) тут ЗАБОРОНЕНІ (1С не дозволяє батч усередині дужок підзапиту).

**Режим 2 (пакетний, з temp-таблицями).** Коли потрібні тимчасові таблиці
(напр. `ИНДЕКСИРОВАТЬ ПО` для швидкості з'єднань, або той самий проміжний
результат джойниться кілька разів) — будь-яка кількість підготовчих пакетів,
а фінальний пакет обов'язково завершується записом у службову
`ТТ_РЕЗУЛЬТАТЗАПИТУ_`:

```
ВЫБРАТЬ ... ПОМЕСТИТЬ ТТ_Проміжна ИЗ ... ;

ВЫБРАТЬ
    Т.Номер       КАК Номер,
    Т.Контрагент  КАК Контрагент
ПОМЕСТИТЬ ТТ_РЕЗУЛЬТАТЗАПИТУ_
ИЗ ТТ_Проміжна КАК Т
```

Роутер сам доклеює фінальну проєкцію:
`ВЫБРАТЬ {fields} ИЗ ТТ_РЕЗУЛЬТАТЗАПИТУ_ ГДЕ {filters} УПОРЯДОЧИТЬ ПО {order}`.
Розпізнавання режиму — за наявністю `ПОМЕСТИТЬ ТТ_РЕЗУЛЬТАТЗАПИТУ_`.

Два обовʼязкові правила Режиму 2:
1. **Завершуй саме на `ПОМЕСТИТЬ ТТ_РЕЗУЛЬТАТЗАПИТУ_`.** Без цього роутер
   піде звичайною гілкою і батч тихо зламається.
2. **Імена колонок `ТТ_РЕЗУЛЬТАТЗАПИТУ_` = публічний контракт виходу.** 1С
   будує рядки відповіді за іменами колонок результату, тож ці імена стають
   ключами JSON у `rows` — вони мають збігатися з аліасами у `fields` .json.
   Інші імена → фронт мовчки отримає порожні поля.

Що бере на себе роутер (памʼятати не треба): хвостовий `;` наприкінці `.sel`
зрізається; `fields` не передані → `ВЫБРАТЬ * ИЗ ТТ_РЕЗУЛЬТАТЗАПИТУ_`;
`offset`/`limit` застосовуються на боці 1С після виконання запиту (нарізка
вже вивантаженої таблиці) — однаково в обох режимах; `total` рахується по
відфільтрованому набору (після `filters`).

Спільне для обох режимів: коментарі `//` на початку рядка loader вирізає,
хвостові коментарі в кінці рядка з кодом — ні (не лишай там `;`); параметри
(`&search` тощо) працюють в обох режимах — один `Запрос` з одним текстом,
`УстановитьПараметр` застосовується до всіх пакетів; `fields` у .json лишай
навіть у Режимі 2 — вона документує очікувані колонки виходу для фронта й MCP.

## Фронтенд — компоненти (html/components/)

Стиль: IIFE-модуль, init(config), працює через CONFIG.API_URL + AUTH (auth.js).
Перед генерацією форми ЧИТАЙ реальний код компонента (read_form), щоб узяти точний API —
нижче лише орієнтир.

- ref_select.js (RefSelect) — універсальний вибір посилання через /1c/query.
  Конфіг: container, query, refKey, displayKey, searchFields, fields[{key,label}],
  onSelect, minChars, matchMode, extraFilter, extraParams. Методи: init, getValue, clear,
  setExtra(id, filter, params), setEnabled(id, bool). CSS: contractor_select.css (класи cs-).
- contractor_select.js (ContractorSelect) — пресет над RefSelect (query=cat_contractors).
  Вимагає ref_select.js ПЕРЕД ним.
- doc_header.js (DocHeader) — шапка документа (сам інжектить CSS, класи dh-).
  Вид/Номер/Дата/Статус/кнопки дій. Дії за станом. Методи: init, setState, getState,
  getDate, setBusy.
- list_view.js (ListView) — універсальний список (таблиця/картки). Дані через /1c/query.
  Конфіг: container, query, columns[{key,label,sortable,width,align}], search{fields,...},
  pageSize, defaultOrder, onRowClick. CSS: list_view.css (класи lv-).

## Фронтенд — глобали

## Форми — як будувати

- Форма будується на ПОЛЯХ ЗАПИТУ (аліасах з .json), не на сирих реквізитах 1С.
- Підключення скриптів — абсолютними шляхами: /html/lib/config.js, /html/lib/auth.js,
  /html/components/<...>. Спершу залежності (ref_select.js), потім пресети.
- Кеш: при зміні JS/CSS версіонуй (?v=N) або Ctrl+Shift+R — стара версія в кеші маскує правки.
- getElementById для guid-id — БЕЗ CSS.escape (він ламає id з провідною цифрою).

## MCP — інструменти (бери контекст, не вигадуй)

Читання: list_objects, describe_object(object_type, object_name), list_queries, get_query(query_name),
  list_forms, read_form(path).
Запис: generate_query(object_type, object_name) — болванка запиту (без запису на диск);
  save_query(sel, meta, file_name); write_form(path, content) — ЛИШЕ pages/ та menu/.
Бекап: create_backup(set_name="full_html") — перед блоком змін.
  Набір full_html тепер: queries1c + html + html_command_log.
Журнал: log_command(cmd, desc, clar, why, files) — запис команди в html_command_log/
  (останній крок блоку змін). user і час ставить сервер; порожній cmd/desc → помилка.

Типові цикли:
- Новий запит: describe_object → generate_query → допрацювати → показати → save_query → log_command.
- Редагувати запит: get_query → змінити ТІЛЬКИ потрібне → save_query (з тим самим file_name!) → log_command.
- Нова форма: read_form кількох наявних (патерн) + get_query (поля) → згенерувати → write_form → log_command.

ВАЖЛИВО про save_query при редагуванні: передавай file_name = реальне імʼя файлу (з get_query.file),
інакше створиться дубль (імʼя файлу може відрізнятися від query_name).

## Журнал команд (html_command_log/)

- Призначення: історія команд користувача (що, з якими уточненнями і чому надходило),
  щоб зберігати контекст між чатами й за потреби аналізувати. Append-only.
- Структура: html_command_log/<user>/<YYYY-MM-DD>/<HHMMSS>_<short-ascii-desc>.md
  (файл на команду; ім'я несе час + короткий ASCII-опис для швидкого перегляду).
- Формат файлу: YAML-frontmatter (time, user, cmd) + тіло (Команда / Уточнення / Чому /
  Зачеплені файли — клікабельні markdown-посилання). Формує бекенд, не ти.
- desc — короткий ASCII-ідентифікатор: літери/цифри/дефіс/підкреслення, без пробілів і
  кирилиці (напр. "currency-form-green"). Українською — вміст полів усередині, не ім'я файлу.
- log_command(files=...): вказувати ЛИШЕ шлях до .sel (не .json) — інакше на одну
  команду save_query створюються два .changes.md сайдкари замість одного.
- Тека входить у бекап full_html (відновлюється разом з проєктом). Свідомо приймаємо:
  відкат html-бекапу відкотить і журнал.
- Пишеться ТІЛЬКИ через log_command (не через Filesystem).

## Обмеження

- write_form пише тільки в pages/ та menu/. lib/, components/, system/ — лише читання.
- Filesystem — тільки читання; запис на диск — виключно через MCP-інструменти.
- Не пиши секрети у файли html/ (вони читаються через API).
- 1С через MCP — тільки читання (даних не змінюємо цим каналом).
- mcp_allowed у метаданих запиту — свідоме обмеження доступу з боку користувача. Claude НЕ встановлює/змінює це поле самостійно (через save_query), навіть маючи технічну можливість — лише за прямою вказівкою користувача.
